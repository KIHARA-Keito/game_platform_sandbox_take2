# PR のレビュー → コメント投稿

対象 PR: `$ARGUMENTS`（番号が渡されていない場合は、`gh pr list --state open` の一覧を提示してどれをレビューするかユーザーに確認してから進める）

このリポジトリは React 18 + TypeScript + Vite + Tailwind CSS v4 のフロントエンドプロジェクト。
このスキルは**レビューコメントの投稿までが役割**。コードの修正・マージ・クローズはユーザーから指示された場合のみ行う。

## 手順

1. **PR の内容を把握する**
   - `gh pr view <番号> --comments` でタイトル・本文・既存のコメントを読む。
   - 本文やブランチ名から元 Issue（`refs #N` や `issue-<番号>-...`）が特定できれば `gh issue view <番号>` で要件を読み、照合できるようにする。
   - 既にこのスキルで投稿したレビューコメントがある場合は再レビューとして扱い、前回指摘の解消状況と新規指摘に絞る。

2. **変更内容を取得して読む**
   - `gh pr diff <番号>` で diff を読み、何を実現しようとした変更かを理解する。
   - diff だけで判断せず、変更されたファイルの全体と呼び出し元・呼び出し先も読む（必要なら `gh pr checkout <番号>` でブランチを取得する。作業ツリーに未コミットの変更がある場合は checkout せず diff とファイル読解のみでレビューする）。

3. **静的チェックを実行する**（PR ブランチを checkout できた場合のみ）
   - `npm run lint` と `npm run build` を実行し、エラー・警告があれば指摘に加える。
   - `package.json` に test スクリプトがあれば `npm test` も実行する。
   - checkout できなかった場合はその旨をレビュー結果に明記する。

4. **変更内容をレビューする**。以下の観点で確認する:
   - **要件との整合**: 元 Issue や PR の説明に対して、実装漏れ・仕様の取り違えがないか
   - **バグ**: ロジック誤り、null/undefined の未処理、境界条件、変更が既存機能を壊していないか（呼び出し元への影響）
   - **React 固有**: hooks の依存配列漏れ、state の直接ミューテーション、クリーンアップ漏れ、key の欠落・不適切な key
   - **TypeScript**: `any` や型アサーションによる型安全性の穴、`src/games/types.ts` のインターフェース違反
   - **プロジェクト規約**: ゲーム本体は `src/games/<ゲーム名>/`、新ゲームは `src/games/registry.ts` に登録、スタイリングは Tailwind ユーティリティクラス、既存の命名・構成との一貫性
   - **スコープ**: PR の目的と無関係な差分（ついでの変更・消し忘れの debug コード・不要ファイル）が混ざっていないか
   - **テスト**: ロジック変更に対応するテストがあるか。なければ「`/add-tests` の実行を推奨」として指摘する（このスキル内でテストは書かない）。
   - 確信の持てない指摘は挙げない。コードを読んで裏付けが取れたものだけを指摘する。

5. **レビュー結果を PR にコメントとして投稿する**
   - `gh pr comment <番号> --body-file <一時ファイル>` で投稿する。本文は一時ファイル経由で渡し、シェルエスケープ崩れを防ぐ。
   - コメントのフォーマット:

     ```markdown
     ## レビュー結果

     （全体サマリ 1〜3 文。問題がなければ「指摘なし」と明言する）

     ### [must] 修正必須（バグ・要件未達・型安全性の破壊）
     - `src/path/to/file.tsx` L10-L20: （指摘内容と修正案）

     ### [should] 修正推奨（規約違反・保守性の問題）
     - ...

     ### [nits] 細かい指摘（好みの範囲）
     - ...

     ### 静的チェック
     - lint: ✅ / ❌（エラー内容）
     - build: ✅ / ❌
     - test: ✅ / ❌ / スクリプトなし
     ```

   - 該当しない重要度のセクションは省略する。各指摘にはファイルパス・行番号・修正案を含める。

6. **結果をユーザーに報告する**
   - 投稿したコメントの URL と指摘の要約（must/should/nits の件数と主な内容）。
   - `gh pr checkout` した場合は元のブランチに戻したことも報告する。
   - コードの修正・再レビュー・マージはユーザーから指示された場合のみ行う。
