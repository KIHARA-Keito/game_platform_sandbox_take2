# フロントエンドの問題発見 → Issues 登録

このリポジトリは React 18 + TypeScript + Vite + Tailwind CSS v4 のフロントエンドプロジェクト。
`$ARGUMENTS` が指定されていればその範囲を、なければ `src/` 全体をレビュー対象とする。

## 手順

1. **静的チェックを実行して機械的に検出できる問題を拾う**
   - `npm run lint`（ESLint: typescript-eslint / react-hooks / react-refresh）
   - `npm run build`（tsc の型チェックを含む）
   - エラー・警告があれば該当箇所を確認し、問題候補に加える。

2. **対象コードを読んでレビューする**。以下の観点でバグ・懸念点を探す:
   - **バグ**: ロジック誤り、null/undefined の未処理、状態更新の競合、境界条件
   - **React 固有**: hooks の依存配列漏れ、不要な再レンダリング、key の欠落・不適切な key、state の直接ミューテーション、クリーンアップ漏れ（イベントリスナー・タイマー）
   - **TypeScript**: `any` の乱用、型アサーションによる型安全性の穴、`src/games/types.ts` のインターフェース違反
   - **アクセシビリティ**: キーボード操作不可、フォーカス管理、alt/aria の欠落、コントラスト
   - **UX/懸念点**: エラー時のフィードバック欠如、ローディング状態の未考慮、レスポンシブ崩れの可能性
   - 確信の持てない指摘は挙げない。実際にコードを読んで裏付けが取れたものだけを問題とする。

3. **既存 Issue との重複を確認する**
   - `gh issue list --state open --limit 100` で既存のオープン Issue を取得し、同じ内容の Issue があれば新規作成せずスキップする（レポートには「既存 #N と重複」と記載）。

4. **発見した問題を Issue として登録する**
   - 1 問題 = 1 Issue。`gh issue create` を使う。
   - タイトル: `[bug]` または `[改善]` プレフィックス + 簡潔な日本語の要約
   - 本文テンプレート:

     ```markdown
     ## 概要
     （何が問題か 1〜3 文）

     ## 該当箇所
     - `src/path/to/file.tsx` L10-L20（該当コードの抜粋）

     ## 影響・再現条件
     （どういう操作・条件で問題になるか）

     ## 修正案
     （具体的な修正方針）
     ```

   - 本文はヒアドキュメントか一時ファイル経由で渡し、シェルエスケープ崩れを防ぐ。

5. **結果をユーザーに報告する**
   - 作成した Issue の番号・タイトル・URL の一覧
   - 重複でスキップしたもの、確信が持てず登録を見送ったものがあればその旨も伝える。
   - 問題が 1 件も見つからなかった場合は Issue を作らず、レビューした範囲と観点を報告して終える。
