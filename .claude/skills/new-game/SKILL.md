# 新規ゲームの自動生成オーケストレーション

お題: `$ARGUMENTS`（渡されていない場合はユーザーに何のゲームを作りたいか確認してから進める）

あなたはこのパイプラインの**オーケストレーター**。自分ではゲームの設計・実装・テスト・レビューをせず、`.claude/agents/` の4つのサブエージェントに委譲して進行管理と品質ゲートの判定だけを行う。各エージェントは Agent ツールで `run_in_background: false`（同期実行）で起動する — 各工程の結果が次の工程の入力になるため。

```
お題 → [game-designer] → 仕様書 → [game-implementer] → 実装
     → [game-playtester] ⇄ 修正ループ → [game-reviewer] ⇄ 修正ループ → コミット
```

## 手順

### 0. 準備

- 作業ツリーに未コミットの変更がある場合は中断してユーザーに報告する。
- main 上で作業しない。設計完了後（id が決まった後）に `git checkout -b game-<id>` を main から作成する。

### 1. 設計 — game-designer

- お題を渡して起動する。プロンプトには「お題」と「仕様書は docs/game-specs/<id>.md に書くこと」を含める。
- 完了したら仕様書を読み、メタ情報（id/title/description/icon）と受け入れ基準があることを確認する。id が既存ゲームと重複していたら設計をやり直させる。
- ここでブランチ `game-<id>` を作成し、仕様の要約（タイトル・ルール1〜2文・受け入れ基準の件数）をユーザーに向けて出力してから次へ進む。ユーザーの承認待ちはしない。

### 2. 実装 — game-implementer

- 仕様書のパスを渡して起動する。
- 完了報告で lint / build が通っていることを確認する。通っていなければ同じエージェントに差し戻す。

### 3. プレイテスト — game-playtester（品質ゲート1）

- 仕様書のパスを渡して起動し、プレイレポートを受け取る。
- **不合格**の場合: プレイレポートの「発見したバグ・懸念」（再現手順付き）を game-implementer に渡して修正させ、再度プレイテストする。この修正ループは**最大2回**。2回修正しても不合格なら、ループを打ち切って残存バグを最終報告に明記する。
- 「未検証（静的検証のみ）」で合格した場合は、その旨を最終報告に含める。

### 4. レビュー — game-reviewer（品質ゲート2）

- 仕様書のパスを渡して起動し、レビュー結果を受け取る。
- **[must] 指摘がある**場合: 指摘一覧を game-implementer に渡して修正させ、再度レビューする。この修正ループは**最大1回**。修正後も must が残る場合は打ち切って最終報告に明記する。
- [should] / [nits] は修正させず、最終報告に記載するだけにする。

### 5. コミットと最終報告

- 仕様書（`docs/game-specs/<id>.md`）とゲーム実装をまとめてコミットする。コミット前に `npm run lint` と `npm run build` を最終確認する。
- ユーザーへの最終報告:
  - ゲームの概要（タイトル・遊び方1〜2文）と起動方法（`npm run dev` → 一覧から選択）
  - 変更ファイル一覧
  - プレイテスト結果の要約（検証方法・受け入れ基準の合否・修正ループの回数）
  - レビュー結果の要約（must/should/nits の件数と、未修正で残した指摘）
  - 残存する既知の問題（あれば）
- push・PR 作成はユーザーから指示された場合のみ行う。

## 運用ルール

- 各工程の間で、どの工程が終わり何が起きたかを1〜2行ずつユーザーに見える形で出力する（サブエージェントの報告はユーザーには見えない）。
- サブエージェントが役割外の作業をした形跡（プレイテスターやレビュアーがコードを変更した等）があれば `git status` で検知し、その変更を revert してから続行する。
- 修正ループの上限を超えても、そこまでの成果はコミットして報告する（lint / build が通らない場合だけはコミットせず、状態と原因を報告して終了する）。
