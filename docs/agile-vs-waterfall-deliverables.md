# アジャイルの場合の設計・成果物の違い

基本設計・詳細設計の成果物一覧はウォーターフォール型を前提としているが、アジャイル(スクラムなど)ではどう違うのかの整理。

## 大きな違い:「工程の成果物」ではなく「継続的に育てるもの」になる

ウォーターフォールでは「基本設計→承認→詳細設計→承認」と工程ごとに完成したドキュメントを納品するが、アジャイルでは**設計は各スプリント内で機能単位に行い、ドキュメントは必要最小限を継続的に更新する**スタイルになる。「基本設計書」「詳細設計書」という名前の成果物自体が存在しない現場が多い。

## ウォーターフォールの成果物がアジャイルでは何に置き換わるか

- **要件定義書・機能一覧** → **プロダクトバックログ**(ユーザーストーリーの一覧)。「〇〇として△△したい、なぜなら□□だから」という形式で、優先順に並べ替え続ける
- **画面設計書・ワイヤーフレーム** → **Figmaなどのデザインカンプ**そのものが仕様になる。紙のドキュメントに転記しない
- **画面項目定義・バリデーション仕様** → ユーザーストーリーの**受け入れ条件(Acceptance Criteria)**に書く。粒度は機能単位で、網羅的な定義書は作らない
- **API詳細仕様書** → **OpenAPI(Swagger)定義**など、コードやツールと同期する「動く仕様」
- **コンポーネント設計書** → **Storybook**やデザインシステムのドキュメント
- **単体テスト仕様書** → **テストコードそのもの**が仕様を兼ねる(受け入れ条件をテストで表現する)
- **方式設計・アーキテクチャ** → これは例外的にアジャイルでも初期(スプリント0)にある程度固める。あとは**ADR(Architecture Decision Record)**で意思決定を都度記録する形が主流

## 考え方の違い

- ウォーターフォールは「実装前に設計を確定させ、承認を得る」= 設計書が**契約・合意の道具**
- アジャイルは「動くソフトウェアが第一の成果物」= ドキュメントは**チームが迷わないための道具**で、書く量は「後で困るかどうか」で判断する
- ただし「アジャイル=ドキュメント不要」は誤解で、**更新され続けない網羅的な文書を作らない**というだけ。API仕様やADRのように長く参照されるものはむしろしっかり管理する

## 実務上の注意

日本のSIer案件では「アジャイルと言いつつ納品物としてウォーターフォール型の設計書一式が契約に含まれる」ハイブリッドな現場も多い。参画時には開発の進め方だけでなく、**契約上の納品ドキュメントが何か**を確認するのが安全。
