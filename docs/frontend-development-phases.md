# フロントエンド開発の工程別対応事項

Web開発でフロントエンドを担当する際の、各工程(要件定義、基本設計、詳細設計、製造、テスト、運用、保守)で対応すべきことの整理。本文は受託・SIer型のウォーターフォール工程を前提とし、各工程末尾の引用ブロックで自社プロダクト開発の文脈ではどう変わるかを注釈している。

## 要件定義

「システムで何を実現するか」を発注者・利用者と合意する工程。フロントエンド担当が最上流から入るケースは多くないが、UI に直結する要求はこの段階で把握しておくと後工程が楽になる。

- **業務要件・ユーザー要求のヒアリング**: 誰が・いつ・何のために使う画面なのか、業務フローとあわせて把握する
- **機能要件の洗い出し**: システムとして必要な機能の一覧化。フロント視点では「どんな操作・画面が必要になりそうか」の当たりを付ける
- **非機能要件の定義**: 性能(同時アクセス数、レスポンス目標)、可用性、セキュリティ、アクセシビリティなどの要求レベルを合意する
- **利用環境の確認**: 対応ブラウザ・OS・デバイス(PC/スマホ/タブレット)の範囲、社内システムなら端末環境の制約を確認する
- **スコープと優先度の合意**: 今回リリースに含める機能と含めない機能の線引き、フェーズ分けの方針を決める
- **要件定義書の作成・レビュー**: 合意内容をドキュメント化し、発注者の承認を得る(以降の設計・見積もりの根拠になる)
- **概算見積もり・体制検討への協力**: 画面数や複雑度からフロントエンド工数の概算を出す

> **自社プロダクト開発では**: この工程は PRD(Product Requirements Document)に相当する。PM が「なぜ作るのか・誰のどんな課題を解決するのか」を背景・成功指標(KPI)・ユーザーストーリー中心に書き、契約文書ではなくチームの認識合わせのための生きたドキュメントとして開発中も更新され続ける。フロントエンドエンジニアは実現性の観点からレビューに参加することが多い。

## 基本設計(外部設計)

「ユーザーやシステム外部から見てどう動くか」を決める工程。

- **画面一覧・画面遷移図の作成**: どんな画面が存在し、どの操作でどこへ遷移するかを定義する
- **ワイヤーフレーム / 画面レイアウト設計**: 各画面の大まかな構成(ヘッダー、ナビ、コンテンツ配置)を決める。デザイナーがいる場合はFigmaなどのデザインカンプとの整合を取る
- **UI/UX方針の決定**: レスポンシブ対応の範囲、対応ブラウザ・デバイス、アクセシビリティ要件(WCAG準拠レベルなど)
- **機能一覧の整理**: 画面ごとに持つ機能(検索、登録、バリデーションなど)の洗い出し
- **API連携の概要設計**: バックエンドとどんなデータをやり取りするか、インターフェースの大枠をバックエンド担当と合意する
- **非機能要件の確認**: 表示速度の目標、セキュリティ方針(認証方式、CSRF対策など)、多言語対応の有無
- **技術選定**: フレームワーク(React/Vue/Next.jsなど)、状態管理、CSS設計方針の決定(基本設計かそれ以前に確定させる)

> **自社プロダクト開発では**: 独立した「基本設計書」を作ることは少なく、Figma などのデザインカンプ+デザインドック(Design Doc)がこの工程の成果物に相当する。画面遷移やレイアウトはデザイナーと直接やり取りしながら決め、技術選定やアーキテクチャ上の大きな判断は ADR(Architecture Decision Record)として残すチームが多い。

## 詳細設計(内部設計)

「開発者がそのまま実装に入れるレベル」まで落とし込む工程。

- **コンポーネント設計**: 画面を構成するコンポーネントの分割・階層・再利用方針(Atomic Designなどの採用有無)
- **画面項目定義**: 各入力欄・ボタンごとの型、桁数、必須/任意、初期値、活性/非活性条件
- **バリデーション仕様**: 入力チェックのルール、エラーメッセージの文言と表示位置・タイミング
- **API仕様の詳細化**: エンドポイントごとのリクエスト/レスポンス定義、エラーレスポンス時の画面挙動
- **状態管理設計**: どのデータをどこで持つか(グローバルストア / コンポーネントローカル / URLパラメータ)
- **画面遷移の詳細**: 遷移時のパラメータ受け渡し、ローディング表示、認証切れ時の挙動
- **異常系の設計**: 通信エラー、タイムアウト、二重送信防止などの挙動定義
- **ディレクトリ構成・命名規約**: コーディング規約とあわせてチームで統一

> **自社プロダクト開発では**: 網羅的な詳細設計書は書かず、デザインドックに主要な設計判断だけを記し、残りはコードと PR(プルリクエスト)そのもので表現するのが主流。画面項目定義やバリデーション仕様のような細部は、Storybook・型定義(TypeScript)・スキーマ(OpenAPI/zod など)が実質的な仕様書の役割を果たす。

## 製造(実装)

- **コーディング**: 詳細設計に基づくコンポーネント・画面の実装
- **単体テストコードの作成**: Jest / Vitest + Testing Library などでのユニットテスト(製造工程に含めることが多い)
- **静的解析・フォーマットの適用**: ESLint、Prettier、TypeScriptの型チェックをCIに組み込む
- **コードレビュー**: プルリクエストベースでのレビュー運用
- **モック対応**: バックエンドが未完成の場合、MSWやモックサーバーでAPIをスタブ化して先行開発
- **進捗管理**: 設計との乖離が出たら設計書へフィードバックして更新する(ここを怠ると保守で苦しむ)

> **自社プロダクト開発では**: 実装内容自体は大きく変わらないが、進め方が異なる。ウォーターフォールの「製造工程」として区切らず、スプリント単位で設計〜実装〜リリースを回す。フィーチャーフラグで未完成機能を隠したまま main ブランチへ頻繁にマージし(トランクベース開発)、継続的デリバリーで小さく出すのが典型。

## テスト

フロントエンド担当として関わる主なテストレベル。

- **単体テスト**: 関数・コンポーネント単位の動作確認(製造と並行が一般的)
- **結合テスト**: 実際のAPIと繋いだ画面動作、画面間遷移、データの受け渡しの確認
- **システムテスト(総合テスト)への対応**: シナリオテストで検出されたフロント起因の不具合修正
- **クロスブラウザ / デバイステスト**: 対応ブラウザ・スマホ実機での表示崩れ、動作差異の確認
- **E2Eテスト**: Playwright / Cypress などでの主要シナリオ自動化
- **非機能テスト**: Lighthouseなどでの表示速度計測、アクセシビリティチェック
- **テスト成果物の作成**: テスト仕様書、エビデンス(スクリーンショット等)、不具合票の起票と管理

> **自社プロダクト開発では**: 独立した「テスト工程」やテスト仕様書・エビデンス作成は薄く、CI で回る自動テスト(ユニット・E2E)が品質担保の中心になる。加えて、社内メンバーによるドッグフーディング、カナリアリリースや段階的ロールアウトで「本番で小さく試して検知する」アプローチを併用することが多い。QA 専任者がいる場合も、工程の門番ではなく開発と並走する形が一般的。

## 運用

リリース後にシステムを安定稼働させる工程。

- **リリース作業**: ビルド・デプロイの実施、リリース手順書の整備、切り戻し手順の準備
- **監視**: Sentry等でのフロントエンドエラー監視、Core Web Vitalsなどパフォーマンスの継続計測
- **問い合わせ対応**: 「画面が表示されない」等の調査(ブラウザ依存か、データ起因か、バックエンド起因かの切り分け)
- **アクセス解析**: GA等の計測タグの維持管理、計測要件の追加対応
- **証明書やドメイン、CDN設定の管理**(インフラチームと分担する場合も多い)

> **自社プロダクト開発では**: 「作る人と運用する人」を分けず、開発チーム自身が運用まで担う(You build it, you run it)のが基本。エラー監視や Core Web Vitals の計測は同じだが、それに加えて A/B テストや機能の利用状況分析など「リリース後にプロダクトを改善するための計測」の比重が大きい。リリースは特別なイベントではなく、日常的に何度も行うものになる。

## 保守

不具合修正や小規模な改善を継続する工程。

- **不具合修正**: 障害調査、原因特定、修正、リグレッションテスト
- **ライブラリ・フレームワークのアップデート**: 依存パッケージの脆弱性対応(npm audit / Dependabot)、フレームワークのバージョンアップ計画
- **ブラウザアップデート追従**: 新バージョンのブラウザでの動作確認、非推奨APIの置き換え
- **小規模な機能改修**: 文言変更、UI改善、項目追加などの継続対応
- **ドキュメント保守**: 設計書・手順書を実態に合わせて更新し続ける
- **技術的負債の返済**: リファクタリング計画の提案・実施

> **自社プロダクト開発では**: 「保守」という独立したフェーズや保守契約は存在せず、不具合修正・依存更新・小改修はすべて通常のプロダクト開発イテレーションの一部として扱われる。バックログ上で新機能と技術的負債の返済が優先度を競う形になるため、負債返済の時間をどう確保するか(例: 稼働の一定割合を改善に充てる)がチーム運営上の論点になる。

## 実務でつまずきやすいポイント

- **基本設計と詳細設計の境界は現場ごとに違う**ので、参画時に「どこまでが基本設計の成果物か」を必ず確認する
- **API仕様の合意が遅れるとフロントは全工程が詰まる**ので、早い段階でバックエンド担当とインターフェースを固めるのが最重要
- 運用・保守では**再現手順の聞き取りと切り分け能力**が問われる。ブラウザのDevTools(Network / Consoleタブ)での調査に慣れておくと強い
