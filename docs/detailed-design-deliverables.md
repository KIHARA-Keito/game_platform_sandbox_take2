# 詳細設計の成果物一覧

一般的なシステム開発(特にウォーターフォール型)における詳細設計の成果物の整理。基本設計が「外から見た振る舞い」を定義するのに対し、詳細設計は「開発者がそのまま実装に入れるレベル」まで内部構造を落とし込む工程であり、成果物もそれに対応したものになる。

## 中核となる成果物

**詳細設計書(内部設計書)**が最終的な成果物になるのが一般的で、以下のドキュメント群の集合を指す。基本設計書と異なり、読者は顧客ではなく**実装者・テスト担当者**である。

### 画面まわり(フロントエンドに最も関係が深い)

- **画面項目定義書**: 画面上の全項目(入力欄・ボタン・表示項目)ごとの型、桁数、必須/任意、初期値、活性/非活性条件、表示フォーマット
- **入力チェック仕様書(バリデーション定義)**: 単項目チェック・相関チェックのルール、チェックの実行タイミング、エラーメッセージの文言と表示位置
- **イベント定義(画面アクション仕様)**: ボタン押下・フォーカス移動などのイベントごとの処理内容、確認ダイアログの有無、二重送信防止などの制御
- **画面遷移詳細**: 遷移時のパラメータ受け渡し、ローディング表示、認証切れ・権限なし時の挙動
- **メッセージ一覧**: エラー・警告・確認メッセージのID・文言・表示方法の一覧

### プログラム構造まわり

- **コンポーネント設計書 / モジュール構成図**: コンポーネントの分割・階層・責務・再利用方針(フロントエンドではAtomic Designなどの構成方針を含む)
- **クラス図・シーケンス図**: 処理の呼び出し関係や時系列の流れ(UMLで書く現場もあれば省略する現場もある)
- **処理フロー(プログラム処理仕様書)**: 機能ごとの処理ロジックをフローチャートや擬似コードで記述したもの
- **状態管理設計書**: どのデータをどこで持つか(グローバルストア / コンポーネントローカル / URLパラメータ)、ストアの構造定義

### インターフェースまわり

- **API詳細仕様書(IF項目定義書)**: エンドポイントごとのリクエスト/レスポンスの項目単位の定義(型、必須、桁数)、HTTPステータスごとのエラーレスポンスと画面側の挙動
- **外部連携詳細仕様**: ファイル連携やメッセージ連携がある場合のレイアウト定義

### データまわり

- **テーブル定義書(物理設計)**: カラム単位の型・制約・インデックスまで確定させたもの(主にバックエンド側の成果物)
- **CRUD図**: 機能×テーブルでどの操作(作成/参照/更新/削除)が発生するかのマトリクス

### 異常系・共通仕様まわり

- **例外処理設計(エラーハンドリング仕様)**: 通信エラー、タイムアウト、想定外エラー時の共通的な挙動とログ出力方針
- **共通部品設計書**: 共通コンポーネント・共通関数・ユーティリティの仕様
- **コーディング規約 / 命名規約 / ディレクトリ構成**: 詳細設計までに確定させてチームで統一するもの

### テストへの引き継ぎ

- **単体テスト仕様書(テスト観点一覧)**: 詳細設計の内容を基にした単体テストのケース定義。詳細設計工程で作成する現場と、テスト工程で作成する現場がある

## フロントエンド担当として特に押さえるべきもの

フロントエンド担当が主に**作成する側**になるのは「画面項目定義書・入力チェック仕様書・イベント定義・コンポーネント設計書・状態管理設計書」あたり。「API詳細仕様書」はバックエンド担当と**共同で定義・合意する**ことが多く、ここの確定が遅れるとフロントの実装全体が詰まるため最優先で固める。

## 補足

- 詳細設計書は「これを読めば設計者以外でも実装できる」レベルが品質基準。逆に、コードを読めば分かることまで書きすぎると保守されない設計書になるため、粒度の見極めが重要
- 実装中に設計との乖離が出たら**設計書側を更新する**運用をチームで徹底しないと、保守工程でドキュメントが信用できなくなる
- モダンな現場では、API詳細仕様をOpenAPI(Swagger)、コンポーネント仕様をStorybookで代替するなど、**動くドキュメント**に寄せるケースも増えている
