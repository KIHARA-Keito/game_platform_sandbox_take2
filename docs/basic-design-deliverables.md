# 基本設計の成果物一覧

一般的なシステム開発(特にウォーターフォール型)における基本設計の成果物の整理。現場によって名称や粒度は変わるが、代表的なものをまとめる。

## 中核となる成果物

**基本設計書**が最終的な納品物になるのが一般的で、以下のドキュメント群をまとめたもの(または個別ドキュメントの集合)を指す。

### 画面まわり(フロントエンドに最も関係が深い)

- **画面一覧**: システム内の全画面をID・画面名・概要付きで列挙したもの
- **画面遷移図**: 画面間の遷移関係を図示したもの
- **画面レイアウト(画面設計書 / ワイヤーフレーム)**: 各画面の構成要素と配置。項目の詳細定義は詳細設計に回し、基本設計では「何がどこにあるか」レベルまでが多い
- **画面機能一覧**: 各画面が持つ機能(検索、登録、CSV出力など)の一覧

### 機能・処理まわり

- **機能一覧**: システム全体の機能を洗い出した一覧表
- **業務フロー図**: システム導入後の業務の流れ(As-Is / To-Be)
- **バッチ処理一覧・帳票一覧**: 画面以外の処理や出力物がある場合

### インターフェースまわり

- **外部インターフェース設計書(IF一覧)**: 外部システムやバックエンドAPIとの連携の概要(何を・いつ・どんな方式でやり取りするか)
- **API一覧(概要レベル)**: エンドポイントと役割の一覧。項目単位の詳細は詳細設計で定義することが多い

### データまわり

- **ER図 / テーブル定義(概要)**: データベースの論理設計。フロント担当が書くことは少ないが、レビューには関わる
- **コード定義(区分値一覧)**: ステータスや種別などのコード体系

### 方式・非機能まわり

- **方式設計書(アーキテクチャ設計書)**: 採用技術、システム構成図、フレームワークの構成方針
- **非機能要件定義との対応表**: 性能目標、可用性、セキュリティ方針(認証・認可方式など)
- **対応環境定義**: 対応ブラウザ・OS・デバイスの範囲

## フロントエンド担当として特に押さえるべきもの

上記のうち、フロントエンド担当が主に**作成する側**になるのは「画面一覧・画面遷移図・画面レイアウト・画面機能一覧」あたりで、「API一覧・ER図・方式設計」は**レビューや協議に参加する側**になることが多い。

## 補足

- 基本設計書は**顧客がレビュー・承認する対象**であることが多いため、「顧客が読んで動きをイメージできるか」が品質基準になる。実装者向けの内部的な話(コンポーネント分割など)は詳細設計に回す
- どこまでを基本設計の成果物とするかは**契約や現場の慣習で大きく変わる**ので、参画時に成果物一覧(WBSや納品物リスト)を確認するのが確実
- アジャイル寄りの現場では、これらを網羅的なドキュメントにせず、Figmaのデザインカンプ+画面遷移図+API仕様(OpenAPI)で代替するケースも増えている
